    28→# 1. Résumé exécutif
    29→
    30→OGHOTEL est une plateforme SaaS multi-tenant destinée aux propriétaires, gérants et responsables d’hôtels, résidences meublées, auberges et autres structures d’hébergement en Côte d’Ivoire.
    31→
    32→L’application centralise dans une interface simple :
    33→
    34→- la gestion des chambres ;
    35→- la gestion des réservations ;
    36→- le calendrier de disponibilité ;
    37→- les check-in et check-out ;
    38→- les fiches clients ;
    39→- les paiements ;
    40→- les factures et reçus ;
    41→- les dépenses ;
    42→- le personnel ;
    43→- le ménage ;
    44→- la maintenance ;
    45→- les rapports et statistiques.
    46→
    47→Le modèle commercial repose sur trois formules d’abonnement annuel :
    48→
    49→- **ESSENTIEL — 30 000 FCFA/an** ;
    50→- **PRIVILÈGE — 50 000 FCFA/an** ;
    51→- **PREMIUM — 75 000 FCFA/an**.
    52→
    53→Le processus de vente est volontairement semi-manuel au lancement :
    54→
    55→1. le prospect visite la landing page ;
    56→2. il remplit un formulaire de demande ;
    57→3. le Super Admin reçoit la demande dans son tableau de bord ;
    58→4. le Super Admin contacte le prospect par WhatsApp ou appel ;
    59→5. le prospect paie par Mobile Money ou autre moyen convenu ;
    60→6. le Super Admin valide le paiement ;
    61→7. le Super Admin génère un code d’activation unique ;
    62→8. le client utilise ce code pour créer son compte administrateur d’hôtel/résidence.
    63→
    64→OGHOTEL doit être professionnel, mais suffisamment simple pour être utilisé par une personne peu habituée aux logiciels de gestion.

    68→# 2. Contexte et problématique
    69→
    70→De nombreux hôtels et résidences en Côte d’Ivoire gèrent encore leurs activités avec :
    71→
    72→- des cahiers papier ;
    73→- des fichiers Excel ;
    74→- des conversations WhatsApp ;
    75→- des appels téléphoniques ;
    76→- des reçus manuels ;
    77→- une mémoire humaine non centralisée.
    78→
    79→Cette organisation crée plusieurs difficultés :
    80→
    81→- risque de double réservation ;
    82→- manque de visibilité sur les chambres disponibles ;
    83→- erreurs dans les calculs de nuitées, acomptes et soldes ;
    84→- absence d’historique fiable des clients ;
    85→- difficulté à suivre les paiements en espèces et Mobile Money ;
    86→- manque de statistiques sur le taux d’occupation ou le chiffre d’affaires ;
    87→- contrôle limité sur les employés ;
    88→- perte d’informations entre réception, ménage, comptabilité et direction ;
    89→- facturation peu professionnelle ;
    90→- absence d’outil local abordable et adapté aux usages ivoiriens.
    91→
    92→OGHOTEL apporte une réponse locale, simple et structurée à ces problèmes.

    96→# 3. Vision produit
    97→
    98→## 3.1 Vision
    99→
   100→Faire d’OGHOTEL une solution locale de référence pour la gestion numérique des hôtels et résidences en Côte d’Ivoire, avec un produit simple, fiable, abordable et adapté aux réalités du terrain : Mobile Money, WhatsApp, FCFA, interface française et utilisation sur mobile.
   101→
   102→## 3.2 Proposition de valeur
   103→
   104→OGHOTEL permet à un établissement de :
   105→
   106→- connaître en temps réel les chambres libres, occupées, réservées, en nettoyage ou hors service ;
   107→- éviter les doubles réservations ;
   108→- créer des réservations rapidement ;
   109→- suivre les paiements, acomptes, soldes et impayés ;
   110→- générer des reçus et factures professionnels ;
   111→- suivre les recettes, dépenses et performances ;
   112→- organiser le ménage et la maintenance ;
   113→- limiter les accès du personnel selon les rôles ;
   114→- obtenir des rapports simples pour piloter son activité.
   115→
   116→## 3.3 Positionnement
   117→
   118→OGHOTEL n’est pas une application complexe réservée aux grands hôtels internationaux. C’est un outil local et accessible, conçu pour les petites et moyennes structures hôtelières, résidences meublées et établissements qui veulent professionnaliser leur gestion sans complexité.

   139→

   142→### Responsabilités
   143→
   144→- gérer la plateforme OGHOTEL ;
   145→- consulter les demandes venant de la landing page ;
   146→- contacter les prospects ;
   147→- enregistrer les paiements reçus ;
   148→- générer les codes d’activation ;
   149→- créer, activer, suspendre ou renouveler les comptes clients ;
   150→- suivre les revenus SaaS ;
   151→- gérer les formules et fonctionnalités ;
   152→- consulter les statistiques globales ;
   153→- exporter les données commerciales.
   154→
   155→## 5.2 Prospect / visiteur
   156→
   157→Personne intéressée par OGHOTEL. Elle visite la landing page, consulte les tarifs et remplit le formulaire de demande d’abonnement.
   158→
   159→## 5.3 Admin Hôtel — client final
   160→
   161→Propriétaire ou gérant d’un hôtel/résidence ayant souscrit à OGHOTEL. Il administre son propre établissement.
   162→
   163→### Responsabilités
   164→
   165→- configurer l’établissement ;
   166→- créer les chambres et types de chambres ;
   167→- gérer les réservations ;
   168→- suivre les paiements ;
   169→- créer des comptes employés selon la formule ;
   170→- consulter les rapports ;
   171→- renouveler son abonnement.
   172→
   173→## 5.4 Personnel hôtelier
   174→
   175→Comptes secondaires créés par l’Admin Hôtel, selon la formule souscrite.
   176→
   177→### Rôles possibles
   178→
   179→- **Manager** : supervision opérationnelle ;
   180→- **Réceptionniste** : réservations, clients, check-in/check-out, paiements ;
   181→- **Comptable** : paiements, dépenses, rapports ;
   182→- **Ménage** : suivi de l’état des chambres ;
   183→- **Maintenance** : incidents et réparations.

   520→## 8.3 Processus d’onboarding client
   521→
   522→### Objectif
   523→
   524→Permettre à un client ayant payé de créer son espace hôtel/résidence grâce au code d’activation fourni par le Super Admin.
   525→
   526→### Étapes
   527→
   528→1. Le client va sur la page “Activer mon compte”.
   529→2. Il saisit son code d’activation.
   530→3. Le système vérifie :
   531→   - existence du code ;
   532→   - statut valide ;
   533→   - non-utilisation ;
   534→   - non-expiration ;
   535→   - formule associée.
   536→4. Si le code est valide, le formulaire d’inscription s’ouvre.
   537→5. Le client renseigne :
   538→   - nom du gérant ;
   539→   - nom de l’établissement ;
   540→   - type d’établissement ;
   541→   - adresse ;
   542→   - ville ;
   543→   - téléphone ;
   544→   - email ;
   545→   - mot de passe ;
   546→   - logo optionnel.
   547→6. Le système crée :
   548→   - le tenant / établissement ;
   549→   - le compte Admin Hôtel ;
   550→   - le profil utilisateur ;
   551→   - l’abonnement actif ;
   552→   - les paramètres par défaut.
   553→7. Le code passe au statut `used`.
   554→8. L’utilisateur est redirigé vers son tableau de bord.
   555→
   556→### Critères d’acceptation
   557→
   558→- un code invalide affiche un message clair ;
   559→- un code expiré ne peut pas être utilisé ;
   560→- un code déjà utilisé ne peut pas être réutilisé ;
   561→- un espace établissement est automatiquement isolé ;
   562→- l’Admin Hôtel accède uniquement à ses données.

   969→# 12. Architecture technique recommandée
   970→
   971→## 12.1 Stack
   972→
   973→- **Frontend** : Next.js recommandé, ou React + Vite ;
   974→- **UI** : Tailwind CSS ou CSS moderne ;
   975→- **Backend** : Supabase ;
   976→- **Base de données** : PostgreSQL ;
   977→- **Authentification** : Supabase Auth ;
   978→- **Fichiers** : Supabase Storage ;
   979→- **Fonctions serveur** : Next.js API Routes, Server Actions ou Supabase Edge Functions ;
   980→- **Déploiement** : Vercel ;
   981→- **Versioning** : GitHub privé recommandé.
   982→
   983→## 12.2 Variables d’environnement
   984→
   985→Exemple de variables à prévoir :
   986→
   987→```env
   988→NEXT_PUBLIC_SUPABASE_URL=...
   989→NEXT_PUBLIC_SUPABASE_ANON_KEY=...
   990→SUPABASE_SERVICE_ROLE_KEY=...
   991→SUPER_ADMIN_EMAIL=...
   992→APP_URL=...
   993→WHATSAPP_CONTACT=...
   994→```
   995→
   996→Règles :
   997→
   998→- `NEXT_PUBLIC_SUPABASE_URL` peut être côté client ;
   999→- `NEXT_PUBLIC_SUPABASE_ANON_KEY` peut être côté client uniquement avec RLS bien configuré ;
  1000→- `SUPABASE_SERVICE_ROLE_KEY` doit rester uniquement côté serveur ;
  1001→- aucun secret ne doit être commité dans GitHub ;
  1002→- utiliser `.env.local` en local et variables Vercel en production.
  1003→
  1004→## 12.3 Principe multi-tenant
  1005→
  1006→Chaque table métier doit contenir un identifiant d’établissement :
  1007→
  1008→- `establishment_id`, `hotel_id` ou `tenant_id`.
  1009→
  1010→Le nom recommandé pour garder un vocabulaire général est :
  1011→
  1012→```text
  1013→establishment_id
  1014→```
  1015→
  1016→Règles :
  1017→
  1018→- un utilisateur hôtelier ne voit que les données liées à son établissement ;
  1019→- le Super Admin peut voir les données globales nécessaires à l’administration SaaS ;
  1020→- les politiques RLS doivent être testées avant production ;
  1021→- les requêtes frontend ne doivent jamais contourner la sécurité.
  1022→
  1023→## 12.4 Sécurité
  1024→
  1025→- Supabase Auth pour tous les utilisateurs ;
  1026→- rôles dans une table `profiles` ;
  1027→- RLS activé sur toutes les tables sensibles ;
  1028→- fonctions serveur pour actions critiques : création utilisateur, activation code, génération code ;
  1029→- audit log ;
  1030→- changement obligatoire du mot de passe initial ;
  1031→- rotation des clés exposées ;
  1032→- validation côté client et côté serveur.

  1302→# 14. Règles métier principales
  1303→
  1304→## 14.1 Abonnement et activation
  1305→
  1306→- un prospect peut être converti en client seulement après paiement validé ;
  1307→- un code d’activation doit être unique ;
  1308→- un code d’activation ne peut être utilisé qu’une seule fois ;
  1309→- un code expiré ne peut pas être utilisé ;
  1310→- l’abonnement démarre à l’activation ou à une date définie par le Super Admin ;
  1311→- la date de fin = date de début + durée du plan ;
  1312→- un compte expiré peut être limité, suspendu ou mis en lecture seule selon choix produit ;
  1313→- le Super Admin peut renouveler manuellement.
  1314→
  1315→## 14.2 Réservations
  1316→
  1317→- une chambre ne peut pas avoir deux réservations confirmées sur la même période ;
  1318→- une réservation annulée ne bloque plus la chambre ;
  1319→- une réservation `checked_out` ne peut plus être modifiée sauf par rôle autorisé ;
  1320→- le check-in passe la chambre en occupée ;
  1321→- le check-out passe la chambre en nettoyage ;
  1322→- les conflits de date doivent être vérifiés côté serveur.
  1323→
  1324→## 14.3 Paiements séjour
  1325→
  1326→- le solde = montant total - total paiements ;
  1327→- les paiements doivent garder une référence utilisateur ;
  1328→- un paiement supérieur au solde doit être bloqué ou demander confirmation Admin ;
  1329→- tous les paiements doivent apparaître dans l’historique.
  1330→
  1331→## 14.4 Facturation
  1332→
  1333→- chaque reçu/facture doit avoir un numéro unique par établissement ;
  1334→- un reçu est lié à un paiement ou une réservation ;
  1335→- une facture/reçu annulé doit rester dans l’historique ;
  1336→- les PDF doivent être stockés dans Supabase Storage si générés.
  1337→
  1338→## 14.5 Permissions
  1339→
  1340→- le Super Admin n’est pas rattaché à un établissement ;
  1341→- chaque Admin Hôtel est rattaché à un ou plusieurs établissements selon formule ;
  1342→- les utilisateurs secondaires sont rattachés à un établissement ;
  1343→- les utilisateurs secondaires n’ont accès qu’aux modules autorisés par leur rôle et leur formule.
